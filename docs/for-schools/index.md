## Classroom Package

| Component | What It Gives Instructors |
|---|---|
| Route map | Which labs to assign by student level and course goal |
| Instructor notes | Prerequisites, expected blockers, and review prompts |
| Assessment rubric | What counts as working proof, partial proof, and cleanup proof |
| Safety checklist | Credential, cost, cloud, and secrets review before students provision resources |
| Student templates | README, architecture note, screenshot log, troubleshooting note |
| Review sessions | Optional live walkthroughs, Q&A, or portfolio review |

support depth, live delivery, and whether custom materials are needed.

## Recommended Routes

| Audience | Route |
|---|---|
| Absolute beginners | Linux basics, local validation, GitHub Actions, one local Kubernetes lab |
| Cloud learners | Safety guide, AWS basics, Terraform/OpenTofu, one cleanup-heavy cloud lab |
| Kubernetes learners | Docker packaging, manifests, GitOps, progressive delivery |
| Security learners | Secrets hygiene, Trivy, SBOMs, signing, supply-chain security |
| Portfolio builders | One local lab, one CI/CD lab, one IaC lab, one capstone |

## Instructor Safety Notes

- Students must use their own credentials.
- Never collect student cloud keys.
- Prefer local-first labs before paid cloud work.
- Require budget alerts before cloud projects.
- Require cleanup evidence after cloud projects.
- Treat screenshots as proof only after secrets are removed.

## What To Ask Before A Cohort

1. How many students are in the cohort?
2. What operating systems do they use?
3. Will the course allow cloud accounts, or should labs stay local-first?
4. Which tools are required: AWS, Azure, Docker, Kubernetes, Terraform, GitHub Actions, Jenkins, Argo CD, Trivy?
5. Is the goal learning, portfolio proof, certification support, or internal team enablement?
